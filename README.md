# learnRdashboard

The Hockey R Learning Hub Shiny dashboard, restructured as an app-package
following the **{leprechaun}** scaffold (`leprechaun::scaffold()`), with
the original single `app.R` broken into modules and its content baked
in as real package data.

## Structure

```
learnRdashboard/
├── DESCRIPTION
├── NAMESPACE
├── data/                      # real, lazy-loaded package data (see below)
│   ├── resources.rda           # the curated learning roadmap
│   ├── all_books.rda           # resources, flattened to one row per book
│   ├── datasets_meta.rda       # the ISAR hockey dataset catalogue
│   └── dataset_names.rda       # display-label -> index, for the dataset picker
├── data-raw/                  # scripts that BUILD the files in data/
│   ├── resources.R              # builds resources + all_books
│   └── datasets_meta.R          # loads/cleans ISAR data, builds datasets_meta
├── R/
│   ├── learnRdashboard-package.R    # package-level imports (@import shiny/bslib/ggplot2, etc.)
│   ├── data.R                 # roxygen documentation for the 4 datasets above
│   ├── zzz.R                  # .onLoad() — static asset path, per leprechaun convention
│   ├── fct_helpers.R          # badge_html(), level_badge(), phase_badge_html(), %||%
│   ├── mod_explorer.R         # Tab 1: Explorer      — explorerUI() / explorer_server()
│   ├── mod_learning_plan.R    # Tab 2: Learning Plan  — learning_planUI() / learning_plan_server()
│   ├── mod_hockey_data.R      # Tab 3: Hockey Data    — hockey_dataUI() / hockey_data_server()
│   ├── mod_progress.R         # Tab 4: Progress       — progressUI() / progress_server()
│   ├── ui.R                   # page_navbar() shell, wires up the four module UIs
│   ├── run.R                  # exported run() — shinyApp(ui, server)
│   └── server.R               # wires up the four module servers
├── inst/
│   ├── app/www/                # static JS/CSS assets (empty for now — no custom front end)
│   ├── img/                    # static images (empty for now)
│   └── run/app.R                # standalone dev entry point (pkgload::load_all() + run())
└── man/                         # generated by devtools::document()
```

## Why `data/` + `data-raw/` instead of loader functions

The first pass at this conversion put the `resources` list and
`datasets_meta` catalogue behind memoized loader functions
(`get_resources()`, `get_datasets_meta()`) in `R/data-*.R`, because all
package code has to live directly in `R/` (R doesn't scan subdirectories
there).

This version goes one step further and treats that content as what it
actually is: **static data**, not logic. `resources` and `datasets_meta`
never change at runtime — the original script built them once and reused
them for every session — so they're now pre-built and saved as `.rda`
files in `data/`, exactly the way any R package ships bundled data
(`?LazyData`, `usethis::use_data()`). The scripts that generate them live
in `data-raw/`, following the standard `usethis::use_data_raw()`
convention:

- `data-raw/resources.R` → builds `resources`, `all_books`
- `data-raw/datasets_meta.R` → loads + cleans the `ISAR` datasets, builds
  `datasets_meta`, `dataset_names`

With `LazyData: true` in `DESCRIPTION`, all four objects are just
available by name (`resources`, `all_books`, `datasets_meta`,
`dataset_names`) to any function in the package — no loader calls, no
memoization needed, and the app itself no longer needs `ISAR` installed
at all (only `data-raw/datasets_meta.R` does, to regenerate the data —
that's why `ISAR` moved from `Imports` to `Suggests`).

### Regenerating the data

If the roadmap content changes, or the `ISAR` package ships updated
hockey data:

```r
# from the package root
source("data-raw/resources.R")       # needs: dplyr, usethis
source("data-raw/datasets_meta.R")   # needs: dplyr, ISAR, usethis
```

Each script ends with `usethis::use_data(..., overwrite = TRUE)`, which
re-writes the corresponding `.rda` file(s) in `data/`.

## How the original `app.R` maps to this package

| Original section (line #s)               | New home                          |
|--------------------------------------------|------------------------------------|
| `data("...", package = "ISAR")` + cleaned datasets + `datasets_meta` | `data-raw/datasets_meta.R` → `data/datasets_meta.rda`, `data/dataset_names.rda` |
| `resources` list + `all_books` flatten     | `data-raw/resources.R` → `data/resources.rda`, `data/all_books.rda` |
| `badge_html()`, `level_badge()`, `phase_badge_html()` | `R/fct_helpers.R` |
| `nav_panel("Explorer", ...)` + explorer server logic | `R/mod_explorer.R` |
| `nav_panel("Learning Plan", ...)` + roadmap server logic | `R/mod_learning_plan.R` |
| `nav_panel("Hockey Data", ...)` + dataset/preview/exercise server logic | `R/mod_hockey_data.R` |
| `nav_panel("Progress", ...)` + progress/notes reactiveVals + observers | `R/mod_progress.R` |
| `ui <- page_navbar(...)` shell             | `R/ui.R`                          |
| `server <- function(input, output, session)` | `R/server.R`                    |
| `shinyApp(ui, server)`                     | `R/run.R` → exported `run()`      |

## Namespacing changes

Because Shiny modules give every input/output an automatic ID prefix,
a few things needed small adjustments versus the original monolithic
`app.R`:

- Every `selectInput()`, `textInput()`, `radioButtons()`,
  `actionButton()`, and `uiOutput()`/`plotOutput()`/`DTOutput()` id is
  now wrapped in `ns("...")` inside each module's `*UI()` function.
- In `mod_hockey_data.R`, the `DTOutput()` and `uiOutput()` calls that
  are generated *inside* `renderUI()` (i.e., server-side) use
  `ns <- session$ns` instead of the UI function's `ns <- NS(id)`.
- In `mod_progress.R`, the dynamically-built per-book input ids
  (`prog_<title>`, `note_<title>`) and the Bootstrap collapse
  target/id pair (`notecollapse_<title>`) are wrapped with
  `ns()` where they're used as **HTML element ids**. Reads via
  `input[[paste0("prog_", ...)]]` and `updateSelectInput(session, ...)`
  are left as the plain (unprefixed) name, since `moduleServer()`
  already scopes `input`/`session` to the module — only the ids you
  construct by hand as raw HTML need explicit `ns()`.

## Running the app

```r
# From an installed copy:
learnRdashboard::run()

# From the package source, in development:
pkgload::load_all()
run()

# or just source the dev entry point:
source("inst/run/app.R")
```

## Verified with `R CMD check`

This package builds, installs, loads/unloads cleanly, and documents with
no errors (`devtools::document()` + `R CMD check` were run against it,
using stub `bsicons`/`ISAR` packages for the offline check environment).
Three expected warnings remain, all inherited from the original `app.R`
content rather than being bugs — the resource descriptions and dataset
metadata use em dashes, curly quotes, ellipses, and emoji (🏒 ⛸️ 🥅 📈 💰),
exactly as in the original script. Harmless for running the app; only
relevant if you intend to submit to CRAN, in which case these would need
`\uxxxx` escaping.
